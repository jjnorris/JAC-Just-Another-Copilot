# Intake and clarify

## goal
Turn a request into a task-ready contract without patronizing the user.

## when to use
- At task start.
- Any time new ambiguity appears.

## prerequisites
- Current request text.
- Repo truth for the affected area, if available.

## steps
1. Classify task type, risk level, and likely evidence needs.
2. List explicit requirements and non-goals.
3. Split ambiguity into blocking, non-blocking, and optional recommendation buckets.
4. Record non-blocking assumptions.
5. Issue a clarification request for anything blocking.
6. Restate the ready-to-execute scope.

## pause points
- Before moving into planning if blocking ambiguity remains.

## evidence to collect
- assumption record
- clarification request or resolution
- scope restatement

## stop conditions
- Request remains materially ambiguous.
- Security-sensitive path lacks required details.

## escalation or review path
- Escalate to human review when ambiguity affects approval-zone work.

## final report contract
Report the final scope, recorded assumptions, open questions, and whether execution is safe to begin.
